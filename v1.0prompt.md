## 1. 项目概述

**Living Villages** 是一个 Minecraft mod,将原版均匀散布的村庄改写为 K 聚落 Poisson 聚类,并在聚落之间建立铁路商队网络和每日市场经济。

### 核心架构约束 (不可违反)

```
┌─────────────────────────────────────────────────────────────┐
│  MC Adapter Layer (依赖 MC API)                              │
│  RailPlacer | CaravanEntityManager | NbtVillageStateStore    │
│  BiomeResolverImpl | ConfigLoader | LivingVillagesMod        │
└────────────────────────┬────────────────────────────────────┘
                         │ depends on (import)
                         ▼
┌─────────────────────────────────────────────────────────────┐
│  Core Layer (pure Java 21, zero Minecraft dependency)        │
│  KCenterGenerator | ClusterDetector | NameGenerator          │
│  RegionalGraph | MarketSimulator | CaravanSimulator           │
│  LevelProgression | TickOrchestrator                         │
│  data package: VillageStateStore + all Records                │
└─────────────────────────────────────────────────────────────┘
```

**4 条硬约束**:
1. **V1 (单写)**: `VillageStateStore` 中每个字段有且仅有一个模块写入
2. **V2 (零业务 import)**: Core 模块不 import 本项目其他 Core 模块 (仅可 import `data` 包 + Java 标准库)
3. **V3 (独立可测试)**: Core 模块通过构造假数据可在纯 JUnit 中测试
4. **V4 (可删除)**: 删除任一模块及其写入字段,其余模块编译通过

## 2. 工作模型: 主 Agent + 子 Agent

### 2.1 主 Agent (你) 的职责

作为主 Agent,你的工作是:

1. **读取进度** — 打开 `v1.0tasks/progress.md`,确认哪些模块待完成
2. **按依赖顺序分派** — 按 Phase 0→1→2→3→4→5 顺序,将模块分派给子 Agent
3. **生成子 Agent 上下文** — 为每个子 Agent 编写一份包含以下内容的简报:
   - 该模块的设计规格 (从 `detailed_design.md` 抽取)
   - 该模块的任务清单 (从 `v1.0tasks/<module>.md` 读取)
   - 该模块的测试要求
   - 该模块的依赖模块已完成代码 (如果有)
4. **验证子 Agent 输出** — 收到子 Agent 报告后:
   - 确认任务清单全部打勾
   - 确认测试全部通过
   - 确认 V1-V4 约束未被违反
5. **更新进度** — 在 `v1.0tasks/progress.md` 中勾选完成的 checkbox
6. **处理跨模块问题** — 如果两个模块的接口对齐有误,由主 Agent 协调修正

### 2.2 子 Agent 的工作流程

每个子 Agent 收到模块简报后:

1. 阅读 `detailed_design.md` 中对应模块的 §章节 (包括接口签名、算法骨架、测试策略)
2. 阅读 `v1.0tasks/<module>.md` 的任务清单
3. 阅读已完成依赖模块的源代码 (如果需要理解数据结构)
4. **实现** — 按任务清单逐项完成
5. **测试** — 按测试策略编写并运行 JUnit 测试 (Core 模块) 或 GameTest (Adapter 模块)
6. **报告** — 向主 Agent 汇报: 完成了哪些任务,测试结果,遇到什么问题

### 2.3 每个模块的标准输出

```
模块完成后产生:
  common/src/main/java/com/livingvillages/
    ├── core/<package>/
    │   └── <ModuleName>.java          ← 主源文件
    │   └── <HelperClass>.java         ← 如有内部类拆分为独立文件
    └── adapter/<package>/
        └── <ModuleName>.java           ← Adapter 模块

  src/test/java/com/livingvillages/
    └── core/<package>/
        └── <ModuleName>Test.java       ← 测试文件
```

### Phase 1 — Core 独立模块 (Phase 0 完成后并行)

#### Sub-agent 1A: KCenterGenerator
- **任务文件**: `v1.0tasks/kcenter-generator.md`
- **设计章节**: `detailed_design.md` §3
- **依赖**: data-layer, config
- **关键算法**: K-Center 贪心选择 + Bridson Poisson Disk Sampling
- **纯函数**: `generateKCenters(seed, rangeX, rangeZ, cfg)` → `List<VillageRecord>`

#### Sub-agent 1B: ClusterDetector
- **任务文件**: `v1.0tasks/cluster-detector.md`
- **设计章节**: `detailed_design.md` §4
- **关键算法**: DBSCAN (网格空间索引, eps=250, minPts=1)

#### Sub-agent 1C: LevelProgression
- **任务文件**: `v1.0tasks/level-progression.md`
- **设计章节**: `detailed_design.md` §10
- **关键算法**: 纯阈值判定 (Lv1→4)

### Phase 2 — Core 依赖模块 (Phase 1 完成后并行)

#### Sub-agent 2A: NameGenerator
- **任务文件**: `v1.0tasks/name-generator.md`
- **设计章节**: `detailed_design.md` §5

#### Sub-agent 2B: RegionalGraph
- **任务文件**: `v1.0tasks/regional-graph.md`

- **需先完成**: ClusterDetector (读 clusters[])

### Phase 3 — Core 经济/商队 (串行: Market → Caravan)

#### Sub-agent 3A: MarketSimulator
- **任务文件**: `v1.0tasks/market-simulator.md`
- **设计章节**: `detailed_design.md` §8
- **内部拆分**: CESPricing → GraphHeatDiffusion
- **关键测试**: CES 定价公式 + 热扩散收敛 + 缺粮惩罚

#### Sub-agent 3B: CaravanSimulator

### Phase 4 — Adapter 模块 (Phase 2-3 完成后并行)

#### Sub-agent 4A: RailPlacer
- **任务文件**: `v1.0tasks/rail-placer.md`
- **设计章节**: `detailed_design.md` §7
- **需 MC 环境**: GameTest

#### Sub-agent 4B: CaravanEntityManager

- **设计章节**: `detailed_design.md` §9.4

## 6. 升级规则 (何时询问用户)

**只在以下 4 种情况下暂停并询问用户**。其余所有决策由主 Agent 或子 Agent 自主完成。

### 情况 1: 仓库状态与设计文档矛盾
- 例: `design_doc.md` 说某个字段由模块 A 写入,但已有代码中模块 B 也在写该字段
- **处理**: 报告矛盾 → 询问以哪个为准 → 更新文档或修正代码

### 情况 2: 必需的源文档缺失
- 例: 子 Agent 需要读取 `detailed_design.md` §7 但该章节不存在
- **处理**: 报告缺失 → 询问是否需要补充设计 → 等待后再继续

### 情况 3: 决策会显著改变产品范围
- 例: 实现中发现某个算法在 Minecraft 环境下不可行,需要替换为不同算法
- 例: 性能测试表明某个模块需要完全不同的架构
- **处理**: 报告发现 → 提出替代方案 → 等待批准

### 情况 4: 继续会覆盖已有工作
- 例: 模块已存在大量代码,但新的实现方案需要删除/重写
- **处理**: 列出会被影响的文件 → 询问是否继续 → 等待确认

### 不需要询问的情况 (自主决策)

- 修复编译错误或测试失败
- 调整实现细节 (只要不改变 public 接口)
- 添加辅助方法或内部类
- 优化性能 (只要保持在性能预算内)
- 补充设计文档中未覆盖的边界情况处理
- 添加额外的测试用例
